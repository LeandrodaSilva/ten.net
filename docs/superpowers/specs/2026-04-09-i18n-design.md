# Ten.net i18n — File-Based Internationalization

## Context

Ten.net uses file-based routing where `page.html`, `route.ts`, `layout.html`,
and `document.html` are auto-discovered per directory. The framework lacks i18n
support. Users need a translation system that follows the same file-based
convention — no centralized translation files, no manual language configuration,
and templates remain readable in their original language.

The `www/` marketing site (6 routes, ~150+ translatable strings) serves as the
proof of concept with en-US, pt-BR, and es.

## Design Decisions

| Decision                | Choice                                               | Rationale                                                                      |
| ----------------------- | ---------------------------------------------------- | ------------------------------------------------------------------------------ |
| File convention         | `i18n.{locale}.json` per directory                   | Auto-discovered like `page.html`; locales derived from filenames               |
| Translation inheritance | Hierarchical (root → leaf merge)                     | Matches layout nesting pattern; shared translations (nav, footer) live in root |
| Translation key         | Original text = key (zero HTML modification)         | Templates stay 100% readable; JSON maps text → translation                     |
| URL strategy            | Locale prefix (`/pt-BR/docs`)                        | SEO-friendly; each language has a canonical URL                                |
| Unprefixed URLs         | Content negotiation + hreflang                       | No redirect; serve detected language; crawlers discover variants               |
| Route handler i18n      | Handler receives `locale` + viewEngine resolves text | Maximum flexibility; both approaches available simultaneously                  |

## 1. Translation File Convention

### File Pattern: `i18n.{locale}.json`

Lives alongside `page.html` and `route.ts` in any route directory:

```
app/
  i18n.en-US.json      # Global translations (layout: nav, footer)
  i18n.pt-BR.json
  i18n.es.json
  layout.html
  document.html
  page.html
  route.ts
  docs/
    i18n.en-US.json    # Docs-level translations (inherits root)
    i18n.pt-BR.json
    i18n.es.json
    layout.html
    page.html
    route.ts
    installation/
      i18n.en-US.json  # Page-specific (inherits root + docs)
      i18n.pt-BR.json
      i18n.es.json
```

### Format: Flat JSON

```json
{
  "Build the web, minimally": "Construa a web, minimamente",
  "Get started": "Começar",
  "Home": "Início"
}
```

Keys are the original English text. Values are translations for that locale.

### Language Auto-Detection

The `routerEngine` scans `i18n.*.json` during filesystem walk (same pass as
`route.ts` and `page.html`). Supported locales are the union of all locale
suffixes found — zero manual configuration.

### Hierarchical Merge

For route `/docs/installation` with locale `pt-BR`:

1. `app/i18n.pt-BR.json` (root)
2. `app/docs/i18n.pt-BR.json` (merge — deeper keys override)
3. `app/docs/installation/i18n.pt-BR.json` (merge — deepest wins)
4. Fallback: original text from template

Same algorithm as `findOrderedLayouts()`.

## 2. Translation Mechanism

### Primary: Automatic Text Replacement

The HTML template is **not modified**. The i18n engine extracts text content
between HTML tags and matches against translation keys:

```html
<!-- page.html — completely untouched -->
<h1>Build the web, minimally</h1>
<p>A minimalist web microframework for TypeScript</p>
<a href="/docs">Get started</a>
```

**Safe replacement algorithm:**

1. Split HTML into segments: tags (protected) vs text (translatable)
2. Protected regions: `<script>...</script>`, `<style>...</style>`,
   `<!-- ... -->`, `<tag attributes>`
3. Normalize whitespace in text segments (trim + collapse)
4. Replace longest keys first (avoids partial matches)
5. Preserve original whitespace/indentation

### Escape Hatch: `{{t:key}}text{{/t}}`

For edge cases where automatic matching is insufficient:

- Text with inline HTML tags:
  `<p>Build with <strong>minimal</strong>
  effort</p>`
- Ambiguous short strings that could match in unintended locations
- Keys that differ from the visible text

```html
<p>{{t:intro}}Build the web with <strong>minimal</strong> effort{{/t}}</p>
```

```json
{
  "intro": "Construa a web com esforço <strong>mínimo</strong>"
}
```

### `{{i18n:selector}}` — Language Selector

The only required placeholder addition. Renders navigation links with endonyms
via `Intl.DisplayNames`:

```html
<!-- In template (layout or page) -->
{{i18n:selector}}

<!-- Rendered output for /pt-BR/docs -->
<nav class="i18n-selector" aria-label="Language">
  <a href="/en-US/docs" hreflang="en-US" lang="en-US">English</a>
  <a href="/pt-BR/docs" hreflang="pt-BR" lang="pt-BR" aria-current="true"
  >Português</a>
  <a href="/es/docs" hreflang="es" lang="es">Español</a>
</nav>
```

Only shows locales that exist for the current route.

## 3. URL Routing

### Locale Prefix

The locale is the first URL path segment:

```
/pt-BR/docs/installation  → locale: "pt-BR",  route: /docs/installation
/es/benchmarks             → locale: "es",     route: /benchmarks
/en-US/                    → locale: "en-US",  route: /
/docs                      → locale: auto,     route: /docs
```

The router strips the prefix before matching against registered routes. No
duplicate routes are created.

### Locale Detection (for unprefixed URLs)

Priority order:

1. **Cookie `ten_lang`** — set when user picks language via `{{i18n:selector}}`
2. **`Accept-Language` header** — browser language preference
3. **First available locale** for the route (alphabetical order)
4. **No translation** — original template text used as-is

Unprefixed URLs serve the detected language directly (no redirect) and include
`<link hreflang>` tags pointing to all prefixed variants.

### Cookie Behavior

Every response to a URL with a locale prefix sets
`Set-Cookie:
ten_lang={locale}; Path=/; SameSite=Lax; Max-Age=31536000`. This
means clicking a link in `{{i18n:selector}}` (which points to a prefixed URL)
automatically persists the preference. Subsequent visits to unprefixed URLs use
this cookie.

## 4. Route Handler Integration

The route handler context gains a `locale` property:

```typescript
export function GET(
  req: Request,
  ctx: { params: Record<string, string>; locale?: string },
): Response {
  const { locale } = ctx;
  // Use locale to generate translated server-side content
  return Response.json({ title: "...", bodyHtml: "..." });
}
```

`locale` is `undefined` when no i18n files exist for the route (backward
compatible).

## 5. SEO and HTTP

### Automatic hreflang

Injected in `<head>` for any page with translations:

```html
<link rel="alternate" hreflang="en-US" href="/en-US/docs" />
<link rel="alternate" hreflang="pt-BR" href="/pt-BR/docs" />
<link rel="alternate" hreflang="es" href="/es/docs" />
<link rel="alternate" hreflang="x-default" href="/docs" />
```

### `<html lang="">` Attribute

Automatically set based on the resolved locale:

- `<html lang="en">` → `<html lang="pt-BR">` (regex replacement on rendered
  HTML)
- If no `lang` attribute exists, injects it

### HTTP Headers

| Header             | When                          | Value                                                    |
| ------------------ | ----------------------------- | -------------------------------------------------------- |
| `Content-Language` | Always (when locale resolved) | `pt-BR`                                                  |
| `Vary`             | Unprefixed URLs only          | `Accept-Language`                                        |
| `Set-Cookie`       | User selects language         | `ten_lang=pt-BR; Path=/; SameSite=Lax; Max-Age=31536000` |

## 6. viewEngine Processing Order

```
 1. Load page.html
 2. Nest layouts ({{content}})
 3. Wrap with document.html
 4. Execute route handler → JSON (with locale in context)
 5. Replace {{var}} and {{{var}}} from handler JSON
 6. Resolve {{t:key}}text{{/t}} escape hatches
 7. Apply automatic text translations from i18n.{locale}.json
 8. Resolve {{i18n:selector}}
 9. Set lang="" on <html>
10. Inject <link hreflang> in <head>
11. Inject Tailwind CSS
```

Variable replacement (step 5) runs before translations (steps 6-7) so
translation values can contain `{{var}}` placeholders that are already resolved.

## 7. i18nEngine Module

New file: `src/i18nEngine.ts`

### Public API

```typescript
// Scan i18n.*.json files alongside routes
function scanTranslations(appPath: string): I18nMap;

// Merge translations hierarchically for a route + locale
function mergeTranslations(
  i18nMap: I18nMap,
  routePath: string,
  locale: string,
): Record<string, string>;

// Detect locale from request
function resolveLocale(
  req: Request,
  path: string,
  availableLocales: string[],
): { locale: string | undefined; strippedPath: string };

// Apply translations to rendered HTML (safe text-node replacement)
function applyTranslations(
  html: string,
  translations: Record<string, string>,
): string;

// Render {{i18n:selector}} placeholder
function renderSelector(
  routePath: string,
  currentLocale: string,
  availableLocales: string[],
): string;

// Generate <link hreflang> tags
function renderHreflang(
  routePath: string,
  availableLocales: string[],
): string;
```

### Internal Type

```typescript
// i18nMap[routeDirectory][locale] = { "original text": "translated text" }
type I18nMap = Record<string, Record<string, Record<string, string>>>;
```

## 8. Build System Integration

### AppManifest Addition

```typescript
interface AppManifest {
  routes: EmbeddedRoute[];
  layouts: Record<string, string[]>;
  documentHtml: string;
  assets: Record<string, EmbeddedAsset>;
  i18n: I18nMap; // NEW
  _seed?: Record<string, any[]>;
}
```

The `collectManifest()` function collects `i18n.*.json` files during the build
phase. The `embeddedRouterEngine` reconstructs i18n data from the manifest.

## 9. Dev Warnings

At server startup (dev mode only):

1. Parse all templates for `{{t:key}}` escape hatches
2. Check each key exists in every available locale
3. Scan text content in templates and check against JSON keys
4. Log warnings (non-blocking):

```
⚠ i18n: key "intro" missing in locale "es" for route /docs
⚠ i18n: text "Get started" has no translation in "es" for route /
```

## 11. Proof of Concept: www/ Site

Add translation files for en-US, pt-BR, es to all 6 routes of the marketing
site:

- `/` (home) — hero, features, CTA, performance labels
- `/docs` — introduction, sidebar nav
- `/docs/installation` — installation guide content
- `/docs/routing` — routing documentation content
- `/docs/templates` — templates documentation content
- `/benchmarks` — table headers, labels

The docs route handlers (`route.ts`) use `ctx.locale` to serve translated
Markdown content.

Add `{{i18n:selector}}` to `www/app/layout.html` header for site-wide language
switching.

## 12. Backward Compatibility

- Routes without `i18n.*.json` files work exactly as before — no locale
  detection, no translation, no URL prefix handling
- The `locale` property in handler context is `undefined` when no translations
  exist
- No breaking changes to existing API
